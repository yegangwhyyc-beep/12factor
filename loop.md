# Agent Loop 策略

> 本文件定义 Trae Agent Loop 的行为策略：任务分解、终止条件、子 Agent 调度、错误恢复。
> 运行参数（阈值 / 超时）放在 `.trae/config.toml`；编码细则见 `.trae/rules.md`。
> 冲突优先级：AGENTS.md > loop.md > rules.md > config.toml。

## 1. Loop 总览

```
用户输入
   │
   ▼
[1] 意图识别 ── 判断任务类型
   │
   ▼
[2] 任务分解 ── 大任务拆为子任务
   │
   ▼
[3] 计划生成 ── 输出可执行步骤列表
   │
   ▼
[4] 执行循环 ── 工具调用 → 观察结果 → 决策下一步
   │            │
   │            ├─ 成功 → 继续
   │            ├─ 失败 → 重试 / 降级 / 终止
   │            └─ 需澄清 → 暂停询问用户
   │
   ▼
[5] 自检 ── 跑测试 / lint / diff review
   │
   ▼
[6] 提交 ── 输出结果 / 创建 PR / commit
```

## 2. 任务类型与必经流程

| 任务类型 | 触发关键词 | 必经流程（不可跳过） |
|---------|-----------|-------------------|
| Bug 修复 | "修 bug" / "fix" / "报错" | 复现测试 → 修复 → 全量测试 |
| 新功能 | "加" / "新增" / "实现" | schema → service → API → 测试 |
| 重构 | "重构" / "refactor" | 保持测试绿色 → 小步提交 |
| 代码审查 | "review" / "审查" | 调用 `code-review` skill |
| 文档更新 | "更新文档" / "docs" | 调用 `documentation-update` skill |
| 安全审计 | "安全" / "audit" | 调用 `/security-audit` |

## 3. 任务分解规则

### 3.1 何时分解
- 子任务数 ≤ 3：单 Loop 内顺序执行
- 子任务数 4-7：建议拆为多个 PR
- 子任务数 > 7：**强制暂停**，要求用户确认范围

### 3.2 分解原则
- 每个子任务必须可独立验证（有明确成功条件）
- 子任务之间显式声明依赖关系
- 跨模块改动优先拆分，避免单次提交影响多模块

### 3.3 输出格式
分解后必须输出计划表：

```markdown
## 执行计划

| # | 子任务 | 模块 | 依赖 | 验证方式 |
|---|--------|------|------|---------|
| 1 | 创建 User schema | app/schemas/ | - | pytest tests/test_user_schema.py |
| 2 | 实现 service | app/services/ | #1 | pytest tests/test_user_service.py |
| 3 | 添加 API 路由 | app/api/ | #1,#2 | curl + pytest tests/test_user_api.py |
```

## 4. 终止条件

Loop 必须**主动停止**的情况：

| 条件 | 行为 |
|------|------|
| 同一错误连续重试 ≥ 3 次仍失败 | 停止，输出失败原因和已尝试方案 |
| 测试套件连续 2 次全红 | 停止，请用户介入 |
| 修改文件数 > 10 | 暂停，请用户确认范围 |
| 单次工具调用超时（默认 120s） | 终止该调用，记录日志 |
| 检测到改动触及禁止文件 | 立即停止，回滚，告警 |
| Token 使用 > 80% 上下文 | 触发 `/compact` 或暂停询问 |

## 5. 错误恢复策略

```
工具调用失败
   │
   ├─ 网络错误 / 超时 ────────→ 重试 2 次（指数退避）→ 降级方案
   ├─ 权限不足 ──────────────→ 立即停止，提示用户授权
   ├─ 测试失败 ──────────────→ 分析失败原因 → 修复 → 重跑
   ├─ Lint 失败 ─────────────→ 自动修复 → 重跑
   ├─ 类型检查失败 ───────────→ 自动修复 → 重跑
   └─ 业务逻辑错误 ───────────→ 停止，输出推理过程
```

**禁止**：
- 静默吞掉错误继续执行
- 在错误状态下创建 commit / PR
- 用 `try/except: pass` 让测试变绿

## 6. 子 Agent（Subagent）调度

### 6.1 何时启用子 Agent
- 子任务互相独立（无共享文件改动）
- 单个子任务预计 > 5 分钟
- 用户显式要求并行

### 6.2 调度约束
- 子 Agent 间文件改动不得冲突（用 git worktree 隔离）
- 子 Agent 数量 ≤ 4（避免上下文爆炸）
- 每个子 Agent 必须返回结构化结果：

```markdown
## Subagent Result
- **任务**：xxx
- **状态**：success / failed / partial
- **改动文件**：[a.py, b.py]
- **测试结果**：passed (12/12)
- **未完成项**：[...]
- **需要主 Agent 决策的问题**：[...]
```

## 7. 自检（Self-Check）阶段

提交前必跑，按顺序：

```bash
# 1. 静态检查
uv run ruff check .
uv run ruff format --check .

# 2. 类型检查（如有）
uv run mypy app/

# 3. 全量测试
uv run pytest --cov=app --cov-report=term-missing

# 4. Git 状态检查
git diff --stat
git diff --cached --stat
```

任一步骤失败 → 回到执行循环修复，不得跳过。

## 8. 上下文管理

| 触发条件 | 行为 |
|---------|------|
| Token > 60% | 提示用户可 `/compact` |
| Token > 80% | 自动 `/compact`，保留最近 10 条消息 |
| Token > 95% | 强制停止当前 Loop，要求用户开新会话 |
| 单文件 > 500 行被读入 | 改用 grep / 部分读取，避免全文件入上下文 |

## 9. 与用户的交互边界

### 9.1 必须询问用户
- 涉及禁止文件（`app/core/security.py` 等）
- 需要新增依赖
- 需要修改数据库 schema
- 任务范围超出原始描述
- 删除文件 / 删除测试

### 9.2 可自主决策
- 选择实现方式（符合 rules.md 即可）
- 重命名局部变量
- 添加注释
- 修复 lint 警告

### 9.3 禁止行为
- 静默修改未提及的文件
- 在用户未要求时创建 commit
- 在用户未要求时 push 到远程
- 自行提升权限（如从 suggest 切到 full-auto）

## 10. Loop 输出标准

每次 Loop 结束必须输出：

```markdown
## 任务总结

**原始需求**：<用户输入>
**完成状态**：✅ 完成 / ⚠️ 部分完成 / ❌ 失败

### 改动清单
- `path/to/file.py` (+12 -3)：新增 xxx 功能
- `path/to/test.py` (+45)：补充测试

### 验证结果
- ruff: ✅
- pytest: ✅ (24/24)
- 覆盖率: 87%

### 后续建议（可选）
- 建议补充集成测试
- 建议在 docs/api.md 中更新端点说明
```
